## 1) Core state variables (combat)

### 1.1 Resources / meters

- **Inspiration** (player resource): spent by cards (especially Composition), may be generated by systems.
- **SongHype** (per active song): rises per loop based on LoopScore; drives Vibe.
- **Vibe** (per audience member): “damage / win pressure” resource; increases at Song End, and optionally via direct effects.
- **Stress** (per musician): “pressure / loss pressure”; increases via audience actions / penalties; can be reduced by cards.
- **BreakdownState** (per musician): `None | Shaken` (separate from status effects).

### 1.2 MVP Status Effects (baseline)

- **Flow** (conceptual scope: Song/Band; MVP implementation: band-wide helper)
  - Increases Loop → SongHype conversion each loop.
- **Composure** (scope: Musician)
  - Absorbs incoming Stress before it applies.

## 2) Data model contracts (authoring)

### 2.1 CardDefinition vs CardPayload

- **CardDefinition**: identity + costs + UI + catalog wiring.
- **CardPayload**: mechanics (what the card does).

Contract: a CardDefinition references exactly one CardPayload.

### 2.2 Unified effects list

All card mechanics are expressed as:

- `CardPayload.effects : List<CardEffectSpec>` (polymorphic list; data-only specs).

Implication: new mechanics are added by adding **new CardEffectSpec subclasses** + editor + runtime interpretation.

### 2.3 Status application is “just another effect”

Statuses are applied by:

- `ApplyStatusEffectSpec { StatusEffectSO status; ActionTargetType targetType; int stacksDelta; float delay; }`

Cards reference a **concrete StatusEffectSO variant** (not only a primitive id).

## 3) Runtime pipeline (end-to-end)

### 3.1 Gig start (once per gig)

- Instantiate / bind:
  - Band musicians (each has `BandCharacterStats` and a `StatusEffectContainer`).
  - Audience members (each has `AudienceCharacterStats` and a `StatusEffectContainer`).
  - Deck + Hand controller.
  - UI state.

- Initialize Inspiration to the gig start value.

### 3.2 Song cycle structure (IMPORTANT definition)

For the purpose of “resets per song”, treat a “Song cycle” as:

1) **Between-Songs Action Window** (pre-song actions)  
2) Composition (choose cards / plan)  
3) Performance Loops (LoopScore → SongHype updates)  
4) Song End conversion (SongHype → Vibe)  
5) Audience Turn (applies Stress etc.)

**Song Start boundary** = entering the Between-Songs Action Window for that song cycle.  
This makes it meaningful to play “prep” actions (Flow/Composure) that affect the upcoming performance.

### 3.3 Song Start (enter Between-Songs window)

Do *song-start resets*:

- Set per-song meters:
  - `SongHype = StartingSongHype` (or 0 + seed, per tuning).
- Reset MVP statuses:
  - **Flow = 0**
  - **Composure = 0**
- Handle Breakdown decay:
  - If a musician is `Shaken`, it expires at **end of next Song** (track this explicitly).

### 3.4 Playing an Action card (Between-Songs phase)

**Input**
- Performer: which musician plays the card.
- Target: depends on the card effects’ `ActionTargetType`.

**Execution**
1) Validate timing (ActionTiming).
2) Spend/Generate Inspiration from CardDefinition economy fields (MVP may constrain these).
3) Execute `payload.effects` in order.
4) Move card to discard/exhaust pile (as defined by deck rules).
5) Emit UI/log updates for observability.

### 3.5 Effect execution semantics (MVP slice)

**A) ApplyStatusEffectSpec**
- Resolve targets from `targetType`.
- Apply stacks delta to the target(s)’ status containers.

**B) ModifyVibe (new effect spec for MVP)**
- Resolve targets (audience).
- `AudienceCharacterStats.Vibe += amount` (clamp as needed).

**C) ModifyStress (new effect spec for MVP)**
- Resolve targets (musicians).
- If `amount > 0`: treat as **incoming stress**:
  - absorb via **Composure** first:
    - `absorbed = min(ComposureStacks, amount)`
    - `ComposureStacks -= absorbed`
    - `Stress += (amount - absorbed)`
- If `amount < 0`: reduce stress directly:
  - `Stress = max(0, Stress + amount)` (since amount is negative)

**D) Breakdown check (post-stress)**
- If `Stress >= StressMax` (after Composure absorption):
  - `Cohesion -= 1` (band-wide)
  - `BreakdownState = Shaken`
  - `Stress = ceil(StressMax/2)`

### 3.6 Performance loops

Per loop:
1) Compute `LoopScore`.
2) Convert to `BaseΔSongHype`.
3) Apply Flow amplification:

`ΔSongHype = BaseΔSongHype * (1 + FlowStacks * FlowMultiplier)`

MVP implementation note:
- If Flow is stored on musicians via band-wide helper, define:

`FlowStacks = sum(FlowStacksPerMusician)`  
(or average; choose ONE and treat it as contract for now).

4) Clamp SongHype to `[0..MaxSongHype]`.

### 3.7 Song End conversion → Vibe

At Song End:
- Convert final SongHype + impressions into per-audience Vibe deltas (tunable constants).
- Apply deltas to `AudienceCharacterStats.Vibe`.

### 3.8 Audience Turn (post-song)

- Audience agents perform actions that may:
  - Apply Stress to musicians (use Composure absorption).
  - Apply statuses (future extensions).

## 4) Status system timing

The Status system supports tick timings (e.g., LoopStart/LoopEnd/SongStart/SongEnd), but the combat loop must call them.

**MVP allowance:** if only Flow/Composure exist and they have “reset per song” semantics, you can implement the reset explicitly at Song Start and skip generic ticking initially.

---

## 5) MVP validation kit (cards you should be able to create + test)

Create these **Action** cards via CardEditorWindow:

1) **Add Vibe** — target: single Audience  
   - Effect: `ModifyVibe +X`

2) **Add Stress** — target: Self (musician)  
   - Effect: `ModifyStress +S` (should be partially absorbed by Composure if present)

3) **Remove Stress** — target: Self  
   - Effect: `ModifyStress -R`

4) **Add Flow** — target: All Musicians (band-wide helper)  
   - Effect: `ApplyStatusEffect Flow +F`

5) **Add Composure** — target: Self or All Musicians  
   - Effect: `ApplyStatusEffect Composure +C`

**Expected observations**
- Vibe values change immediately when playing Add Vibe.
- Stress changes immediately; if Composure exists, Add Stress consumes Composure first.
- During performance loops, Flow causes faster SongHype gain (visible in logs/UI).
- If Stress crosses StressMax, Shaken triggers and reduces future Action usage.
